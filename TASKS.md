## Загальний опис застосунку

Ваш сервер буде чистим JSON API без жодного рендерингу HTML. Фронтенд існує окремо і звертається до сервера через HTTP-запити. Такий підхід є стандартом у сучасній розробці, де бекенд обслуговує React-застосунки, мобільні додатки або будь-яких інших клієнтів через єдиний інтерфейс.

Застосунок реалізує дошку оголошень де користувачі можуть публікувати, редагувати та видаляти власні оголошення. Анонімні відвідувачі бачать список оголошень і можуть переглядати деталі кожного з них. Для створення оголошення потрібна реєстрація. Редагувати та видаляти можна лише власні оголошення.

Автентифікація побудована на JWT з refresh токенами. Користувач реєструється, отримує пару токенів і використовує access token для доступу до захищених маршрутів. Коли access token спливає, клієнт отримує нову пару через refresh token без повторного введення пароля.

Ви отримуєте мінімальний бойлерплейт з налаштованими залежностями та конфігурацією OpenAPI. Все інше реалізуєте самостійно.

## Функціональні вимоги

### Auth маршрути

#### `POST /auth/register`

Реєстрація нового користувача. Приймає username, email, password та name у тілі запиту. Створює користувача з хешованим паролем і повертає пару токенів.

Правила валідації:
| Поле      | Вимоги                                      |
|------------|----------------------------------------------|
| `username` | рядок, обов'язковий, 3–30 символів          |
| `email`    | email, обов'язковий                         |
| `password` | рядок, обов'язковий, мінімум 6 символів     |
| `name`     | рядок, обов'язковий, мінімум 2 символи      |


Приклад відповіді:
```json
// 201 Created
{
  "user": {
    "id": 1,
    "username": "ivan_petrenko",
    "email": "ivan@example.com",
    "name": "Іван"
  },
  "accessToken": "eyJhbGci...",
  "refreshToken": "eyJhbGci..."
}
```

Перед створенням перевірте що користувач з таким `username` або `email` ще не існує. Якщо існує, поверніть 409 з повідомленням `"Username or email already taken"`. Пароль зберігається виключно у вигляді bcrypt-хешу. У відповіді поле `password` не повертається ніколи. Access token має термін дії 15 хвилин, refresh token — 7 днів. Refresh token зберігається в базі даних разом з `userId`.

#### `POST /auth/login`

Вхід користувача. Приймає `username` та password, перевіряє їх і повертає нову пару токенів.

Приклад відповіді:
```json
// 200 OK
{
  "user": {
    "id": 1,
    "username": "ivan_petrenko",
    "email": "ivan@example.com",
    "name": "Іван"
  },
  "accessToken": "eyJhbGci...",
  "refreshToken": "eyJhbGci..."
}
```

Якщо користувач з таким `username` не знайдений або пароль невірний, повертайте 401 з однаковим повідомленням `"Invalid credentials"` для обох випадків. Це важливо для безпеки — зловмисник не повинен знати що саме невірне. При успішному вході старий refresh token користувача в базі замінюється новим.

#### `POST /auth/refresh`

Оновлення access токена. Приймає refresh token у тілі запиту і повертає нову пару токенів.

Приклад відповіді:
```json
// 200 OK
{
  "accessToken": "eyJhbGci...",
  "refreshToken": "eyJhbGci..."
}
```
Перевірте підпис токена через jwt.verify(). Знайдіть запис у базі даних за значенням токена. Якщо токен невалідний або такого refresh токена немає в базі, поверніть 401. Після успішного оновлення старий refresh token видаляється з бази і записується новий. Це називається token rotation і захищає від повторного використання викрадених токенів.

#### `POST /auth/logout`

Вихід користувача. Захищений маршрут.

Інвалідує refresh token поточного користувача.
```json
// 204 No Content
```
Знайдіть і видаліть refresh token поточного користувача з бази. Використовуйте deleteMany замість delete, бо якщо токена вже немає в базі, delete викине помилку. Повертайте 204 незалежно від того чи був токен в базі.

#### `GET /auth/me`

Профіль поточного користувача. Захищений маршрут.

Повертає дані автентифікованого користувача.
```json
// 200 OK
{
  "id": 1,
  "username": "ivan_petrenko",
  "email": "ivan@example.com",
  "name": "Іван",
  "createdAt": "2025-01-10T12:00:00.000Z"
}
```

### Маршрути оголошень

#### `GET /announcements`

Список оголошень з пагінацією, пошуком та сортуванням. Публічний маршрут.

Маршрут приймає три опціональні query-параметри:

| Параметр | Опис |
|----------|------|
| `search` | пошук по назві, нечутливий до регістру |
| `sort` | порядок сортування: `newest` (за замовчуванням) або `oldest` |
| `page` | номер сторінки, число більше нуля |

Приклад відповіді:
```json
// 200 OK
{
  "data": [
    {
      "id": 5,
      "title": "Продам ноутбук ASUS",
      "description": "Відмінний стан, 16GB RAM",
      "price": 18000,
      "category": "sale",
      "createdAt": "2025-01-10T12:00:00.000Z",
      "updatedAt": "2025-01-10T12:00:00.000Z",
      "user": {
        "id": 1,
        "username": "ivan_petrenko",
        "email": "ivan@example.com",
        "name": "Іван"
      }
    }
  ],
  "pagination": {
    "total": 23,
    "page": 2,
    "totalPages": 3,
    "perPage": 10
  }
}
```

Пошук шукає підрядок у полі title. Якщо параметр search відсутній або порожній, повертаються всі оголошення. Сортування реалізується через orderBy Prisma: за замовчуванням createdAt: 'desc', якщо `sort === 'oldest'` — createdAt: 'asc'.

Пагінація — 10 записів на сторінку через skip та take. Для підрахунку totalPages використовуйте prisma.announcement.count() з тим самим where-об'єктом що і для findMany().

#### `GET /announcements/:id`

Отримання одного оголошення за ID. Публічний маршрут.

Повертає повний об'єкт оголошення з даними автора.
```json
// 200 OK
{
  "id": 5,
  "title": "Продам ноутбук ASUS",
  "description": "Відмінний стан, 16GB RAM",
  "price": 18000,
  "category": "sale",
  "createdAt": "2025-01-10T12:00:00.000Z",
  "updatedAt": "2025-01-10T12:00:00.000Z",
  "user": {
    "id": 1,
    "username": "ivan_petrenko",
    "email": "ivan@example.com",
    "email": "ivan@example.com",
    "name": "Іван"
  }
}
```
Нагадуємо, що метод `findUnique` повертає null якщо запис не знайдено.

Перевірте результат і поверніть 404 самостійно. Методи update та delete поводяться інакше — вони викидають помилку з кодом P2025, яку error handler перехоплює автоматично.

#### `POST /announcements`

Створення оголошення. Захищений маршрут.

Правила валідації

| Поле | Вимоги |
|-------|---------|
| `title` | рядок, обов'язковий, 5–50 символів |
| `description` | рядок, обов'язковий, мінімум 10 символів |
| `price` | число, обов'язкове, більше нуля |
| `category` | рядок, обов'язковий, одне з: `sale`, `service`, `job`, `other` |

Приклад відповіді
```json
// 201 Created
{
  "id": 6,
  "title": "Продам велосипед",
  "description": "Гірський велосипед, 21 швидкість",
  "price": 8000,
  "category": "sale",
  "createdAt": "2025-01-11T10:00:00.000Z",
  "updatedAt": "2025-01-11T10:00:00.000Z",
  "user": {
    "id": 1,
    "username": "ivan_petrenko",
    "email": "ivan@example.com",
    "name": "Іван"
  }
}
```
`userId` береться з токена через `req.user.sub`, а не з тіла запиту. Клієнт не може вказати автора вручну.


#### `PATCH /announcements/:id`

Часткове оновлення оголошення. Захищений маршрут з перевіркою ownership.

Правила валідації ті самі, що при створенні, але всі поля опціональні. При цьому хоча б одне поле має бути присутнє в тілі запиту — порожній `{}` не є валідним запитом.

```json
// 200 OK
{
  "id": 6,
  "title": "Продам велосипед терміново",
  "description": "Гірський велосипед, 21 швидкість",
  "price": 6500,
  "category": "sale",
  "createdAt": "2025-01-11T10:00:00.000Z",
  "updatedAt": "2025-01-12T14:30:00.000Z",
  "user": {
    "id": 1,
    "username": "ivan_petrenko",
    "email": "ivan@example.com",
    "name": "Іван"
  }
}
```
Перед оновленням перевірте що оголошення належить поточному користувачу.

Порівняйте announcement.userId з req.user.sub. Якщо не збігається, поверніть статус 403 з повідомленням `"Access denied"`.


#### `DELETE /announcements/:id`

Видалення оголошення. Захищений маршрут з перевіркою ownership.
```json
// 204 No Content
```
Перевірка ownership така сама як і при оновленні. Після видалення використовуйте `res.status(204).end()`. Метод `end()` завершує відповідь без тіла — статус 204 за специфікацією HTTP не повинен мати тіла відповіді.


## Технічна реалізація

### Prisma schema

Ваша схема має містити три моделі.

**Модель `User`** зберігає дані користувача: унікальний username, хешований password, name та дату реєстрації. Модель має зв'язки з оголошеннями та refresh токенами.

**Модель `RefreshToken`** зберігає токен та зв'язок з користувачем. Поле токена має бути унікальним для пошуку в базі.

**Модель `Announcement`** містить заголовок, опис, ціну, категорію та обов'язковий зв'язок з автором. Поля createdAt та updatedAt відстежують час створення та останньої зміни.

⚠️Після визначення схеми запустіть міграцію.

## Підказки для реалізації

Рекомендуємо почати з auth маршрутів. Вони не залежать від оголошень, і після їх реалізації ви зможете отримувати токени для тестування захищених маршрутів. Потім переходьте до CRUD оголошень, спочатку публічні GET маршрути, потім захищені. Ownership перевірку додавайте в останню чергу.

Middleware автентифікації витягує токен з заголовка Authorization, верифікує його і кладе декодовані дані в req.user. Щоб TypeScript знав про поле req.user, потрібна module augmentation — розширення типу Request через declare global.

Для валідації створіть три middleware: validateBody, validateParams, validateQuery.

Кожен приймає Zod схему і повертає Express middleware. Використовуйте safeParse() щоб не кидати виняток при не-валідних даних.

Ownership перевірка відбувається в контролері. Спочатку знайдіть оголошення, перевірте що воно існує, потім порівняйте announcement.userId з req.user.sub.

Для реєстрації `bearerAuth` в OpenAPI документації використовуйте `registry.registerComponent()`.

Поле updatedAt в Prisma schema оновлюється автоматично при використанні директиви @updatedAt:

```js
createdAt DateTime @default(now())
updatedAt DateTime @updatedAt`
```

Змінні середовища для JWT:
```js
JWT_SECRET=your_secret_key_here
```
